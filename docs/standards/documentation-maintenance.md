# 文档维护规则

## 目录组织

`docs/` 按系统域组织，不再在根目录使用 `00/01/02` 编号文件维持顺序。

- `product/`：产品定位、需求、交互、UI 和路线图。
- `architecture/`：架构总览、模块边界和阶段性架构治理。
- `data/`：数据源、存储、schema、导入导出和数据模型。
- `guides/`：测试、验收、健康检查和操作型指南。
- `strategy/`：未来阶段、云端化、订阅、Notion 和 AI 策略。
- `standards/`：文档维护和协作规范。
- `adr/`：长期技术决策记录。

## 索引规则

- `docs/README.md` 必须链接每个一级目录。
- 每个一级目录必须有 `README.md`。
- 目录 README 必须说明目录用途、当前有效文档和维护触发条件。
- 移动、重命名或新增文档时，同步更新根索引和对应目录索引。

## 文档类型

参考 Diataxis、GitLab、Microsoft Learn 和 Kubernetes 的文档维护方式，本项目使用轻量分类：

- Concept：解释概念和设计理由，例如架构分层、数据源原则。
- Task：说明如何完成操作，例如测试验收、健康检查。
- Reference：稳定事实和接口说明，例如模块边界、storage schema。
- Strategy：未来阶段方向，必须明确是否属于当前实现范围。

一篇文档可以包含多个章节，但不要让同一页面同时承担所有读者目的。内容膨胀时拆到对应系统域目录。

## 草稿和历史文档

- 设计草稿应晋升到 `product/`、`architecture/`、`data/` 或 `strategy/` 后再作为正式文档维护。
- 未来阶段文档放入 `strategy/` 或明确阶段目录，不与当前验收文档混写。
- 废弃但有决策价值的内容写入 ADR，并标注“已废弃”或“被取代”。

## 变更同步

- 代码变更影响产品行为时，更新 `product/` 和必要验收指南。
- 代码变更影响模块职责、入口、权限或分层时，更新 `architecture/`。
- 数据结构、storage key、导入导出或 migration 变化时，更新 `data/`。
- 验证命令、手动验收或健康检查变化时，更新 `guides/`。
- 持久技术决策变化时，新增或更新 `adr/`。

每轮非轻量工作收尾仍按项目规则写入 `.ai/logs/`。
