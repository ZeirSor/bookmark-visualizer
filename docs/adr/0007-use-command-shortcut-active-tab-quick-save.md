# ADR 0007: 使用扩展命令和 activeTab 实现快捷保存

## 状态

已接受

## 背景

产品需要从任意普通网页快速保存当前页面为浏览器原生书签，并在保存时补充备注和预览图片。用户希望使用类似 `Ctrl + S` 的快捷入口，但 Chrome / Edge 的浏览器和系统快捷键可能优先于扩展命令，不能保证扩展稳定覆盖默认保存页面行为。

项目既有原则是不默认请求广泛 host 权限；网页内容访问应由用户主动触发并按需授权。

## 备选方案

- 强制拦截 `Ctrl + S`：需要长期注入内容脚本或请求广泛站点权限，且仍无法覆盖浏览器受保护页面和部分浏览器保留快捷键。
- 默认使用 `Ctrl + Shift + S` 命令：符合扩展 commands 模型，用户可在浏览器扩展快捷键设置中修改。
- 安装时请求所有站点权限并常驻注入：体验直接，但权限警告和隐私成本过高。

## 决策

第一版快捷保存默认使用 `Ctrl + Shift + S` / macOS `Command + Shift + S` 的扩展命令，不强制覆盖 `Ctrl + S`。

快捷键触发后由 service worker 使用 `activeTab` 和 `scripting` 向当前普通网页注入快捷保存浮框。浮框读取当前页面 URL、标题和候选图片；保存请求回到 service worker，由 `chrome.bookmarks` 创建原生书签，并把备注、预览图片 URL 写入 `chrome.storage.local` 的插件元数据。

注入脚本必须保持可重复执行：运行时代码包在局部闭包中，只通过一个命名的 `window.__bookmarkVisualizerQuickSaveOpen__` 入口暴露重开能力，关闭浮框时清理该入口。这样同一页面连续触发命令时，不会因为内容脚本顶层 lexical 声明重复而在 guard 运行前失败。

自定义快捷键不在应用内部伪造配置，交给 Chrome / Edge 原生扩展快捷键管理页处理。

## 后果

- 不默认请求全站 host 权限，延续主动授权和低权限原则。
- 快捷保存无法在 `chrome://`、扩展商店等不可注入页面直接显示浮框，需要降级提示。
- 用户可以通过浏览器原生快捷键管理修改组合键，但扩展本身不保证所有浏览器/系统保留组合都可用。
- 同一普通网页中关闭快捷保存浮框后再次触发命令，应重新注入或重新打开浮框。
- 保存书签结构仍以浏览器原生 bookmarks API 为唯一事实来源；插件只保存备注和预览图片等补充元数据。
